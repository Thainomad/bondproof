# BondProof — Step-by-Step Build Guide

Companion to `bond-app-mvp-spec.md`. Follow in order. Assumes ~5–10 hrs/week, using Claude Code to write the code. You direct and test; Claude Code implements.

---

## Phase 0 — Setup (1 evening)

1. **Install the tools**
   - Node.js LTS (nodejs.org)
   - Git + a GitHub account
   - Claude Code (desktop app or terminal)
   - VS Code (optional but useful for reading what's built)

2. **Create accounts (all free tiers)**
   - GitHub — code hosting
   - Vercel — app hosting, connects to GitHub
   - Supabase — Postgres database + auth + file storage in one (simplest path; skip separate S3 for MVP)
   - Stripe — payments (activate later, register in test mode now)
   - Resend — transactional email
   - Anthropic Console — API key for the letter-generation feature

3. **Create the project**
   - New GitHub repo: `bondproof`
   - Put both spec files in the repo root: `bond-app-mvp-spec.md` and this guide
   - Open the repo in Claude Code

4. **First Claude Code prompt:**
   > Read bond-app-mvp-spec.md. Scaffold a Next.js App Router project (TypeScript, Tailwind) with Supabase for auth, Postgres and storage. Set up the database schema from section 6 of the spec as SQL migrations. Mobile-first. Don't build features yet — just scaffold, connect Supabase, and get auth (email magic link) working.

5. **Verify:** run `npm run dev`, sign up with your own email, confirm the magic link works. Deploy to Vercel now — deploy from day one, every phase ends with a deploy.

**Done when:** you can log in on your phone via the Vercel URL.

## Phase 1 — Tenancy + photo pipeline (Week 1)

1. Prompt Claude Code:
   > Build tenancy creation per Journey A step 1 in the spec: form for address, lease dates, agent details, bond amount, optional RBO number. One active tenancy per user. Then build the photo upload pipeline: browser camera capture, upload to Supabase storage, extract EXIF (timestamp + GPS) server-side, compute SHA-256 of the original file, store all of it per the photos table. Never strip EXIF. Keep original + generate a web-optimised derivative.

2. **Seed the checklist:** ask Claude Code to create the NSW checklist JSON from spec §4 and a seed script. Review the items yourself — add anything you remember from O'Brien St.

3. **Test on your phone:** create a tenancy, take 5 photos of your current apartment through the app. Check in Supabase that hash, EXIF timestamp and GPS were captured.

**Done when:** photos taken on your phone land in storage with hash + metadata recorded.

---

## Phase 2 — Guided entry capture (Week 2) ⚠️ the make-or-break screen

1. Prompt:
   > Build the guided entry capture flow (Journey A steps 2–3). Room-by-room from the checklist: show item, guidance text, camera button, condition rating (Good/Fair/Damaged), optional note. Must be fast one-handed on a phone: big tap targets, auto-advance to next item after capture, progress indicator, resumable if interrupted. High-claim items prompt for extra photos.

2. **Dogfood brutally:** walk your entire apartment doing a full capture. Time it. Target: whole apartment under 30 minutes. Note every annoyance — each one is a fix for Claude Code.

3. Iterate until the flow feels effortless. This screen decides whether anyone uses the product.

**Done when:** you can document a full apartment in <30 min without frustration.

---

## Phase 3 — Entry report PDF (Week 3) 🎯 deadline: 13 July

1. Prompt:
   > Build entry report PDF generation: server-side HTML-to-PDF (Playwright). Professional layout — property/tenant/agent details, date, then room-by-room: photos with captions, condition ratings, notes, photo timestamps. Footer on every page: capture date + "photos verified by cryptographic hash". Store per the documents table, downloadable and emailable.

2. **THE REAL TEST — Birriga move-in, 13 July:** do your actual entry documentation through the app. This is simultaneously your genuine legal protection for the new lease and the best product research you'll ever do. Bring a charger.

3. Fix everything that annoyed you during the real capture. Attach the generated PDF when you return the agent's condition report (within the 7-day window).

**Done when:** your real entry report exists, generated by your own product.

---

## Phase 4 — Exit capture + comparison (Week 4)

1. Prompt:
   > Build exit capture (mirrors entry flow, same checklist) and the side-by-side comparison view: entry photo vs exit photo per item, condition ratings shown, flag items where rating worsened or that are high_claim_flag. Then exit report PDF (same pattern as entry). Add a pre-handover cleaning checklist screen derived from high-claim categories.

2. **Test with your O'Brien St photos** if you documented anything there — or stage a fake exit capture of one room at Birriga.

**Done when:** comparison view works and exit PDF generates.

---

## Phase 5 — Dispute kit (Week 5) 💰 the paid product

1. Prompt:
   > Build the dispute flow (Journey C): user enters agent's claimed deductions as line items (category, description, amount). For each disputed item generate: (a) a response letter via the Anthropic API — use the template-constrained prompt approach in the spec: cite only evidence items that exist in the DB, reference fair wear and tear where the rules config supports it, low temperature, factual tone, disclaimer footer; (b) an evidence pack PDF — per disputed item show entry vs exit photos, timestamps, notes, then chronology of the tenancy; (c) NCAT application pre-fill using pdf-lib on the current NCAT tenancy application form.

2. **Verify the legal numbers now, not from memory:** open NSW Fair Trading and NCAT sites, confirm the dispute window after a bond claim, NCAT time limits and current fee, and put them in the rules config with citation URLs.

3. **Quality-check the letter generator:** run 5 fake disputes (make up claims against your Birriga evidence). Every letter must reference only real evidence. If it invents anything, tighten the prompt with Claude Code until it can't.

**Done when:** a fake dispute produces a letter, evidence pack and pre-filled NCAT form you'd actually be comfortable sending.

---

## Phase 6 — Money + deadlines + launch (Week 6)

1. **Stripe:** one-off $49 Dispute Kit via Stripe Checkout. Free tier: Journeys A + vault + entry PDF. Paywall sits at the start of Journey C (and the exit comparison). Test in Stripe test mode end to end.
2. **Deadline engine:** deadlines table + daily cron (Vercel cron) that emails via Resend — condition report return reminder at tenancy creation; dispute-window countdown when a dispute opens.
3. **Landing page:** one page, one promise — "Win your bond back. Document your rental properly from day one." Free entry report as the hook. Email capture.
4. **Legal hygiene before real payments:** T&Cs + privacy policy (photo retention, export, delete), disclaimer on all generated documents, ABN sorted (you have the sole trader structure already — check with your accountant whether it sits under it or separate).
5. **Ship:** switch Stripe live. Post your launch in 2–3 Sydney rental groups framed as help, not promo: "I built a free tool to properly document your rental condition at move-in — I'm a renter who nearly lost my bond."

**Done when:** a stranger can sign up, document a property free, and buy a dispute kit.

## Parallel track — validation (weeks 1–3, non-negotiable)

While building, spend 1–2 hrs/week here:

1. Find 3 people mid-bond-dispute (Sydney rental FB groups, r/sydney, friends-of-friends). Help them manually: organise their photos, draft their response, walk them through RBO/NCAT.
2. After each: "Would you have paid $49 for a tool that did this?" Record the answer verbatim.
3. **Kill criteria (from the spec):** if after 10 assists nobody would pay, stop before Phase 5 and reassess. The entry-capture tool still has value as a pivot point; the dispute kit pricing is the hypothesis being tested.

---

## After launch — first 90 days

- **Content = your only channel.** One genuinely useful article/week: "Fair wear and tear vs damage in NSW — with photos", "What to do in the 14 days after a bond claim", "The 12 photos agents wish you hadn't taken at move-in". These rank and compound.
- Answer bond questions in FB groups/Reddit helpfully; link only when directly relevant.
- Watch the funnel: signups → completed entry captures → (months later) dispute kits. Early proxy metric: completed entry captures. If people start but don't finish capture, fix Phase 2 again.
- Feature requests will arrive — hold the line on scope. The scam checker and lease-break calculator are v2, and only after the first paid dispute kits.

---

## Working rhythm with Claude Code

- One phase = one milestone. Don't let it build ahead of the plan.
- Always end a session with: deploy, then test on your **phone** — this product lives on phones.
- When something's wrong, describe the user experience ("capture flow loses my place when the screen locks"), not the suspected code cause. Let it diagnose.
- Commit after every working feature. Ask Claude Code to write tests for the rules engine and hash pipeline — the two places where silent bugs cost users real money.
